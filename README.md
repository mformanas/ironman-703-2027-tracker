# IRONMAN 70.3 Training Tracker

A self-contained web app to track an 77-week training plan for **IRONMAN 70.3
La Quinta (race day: 5 Dec 2027)**. Log swims, bikes, runs and strength sessions,
watch weekly/overall progress, and compare with friends on a shared leaderboard.

**Live site:** https://mformanas.github.io/ironman-703-2027-tracker/

It also installs to an iPhone/Android home screen as a PWA, and can be wrapped as
a native iOS app (see `ios-app/`).

---

## Tech stack

- **Frontend:** a single self-contained `index.html` — vanilla HTML/CSS/JS, no
  framework, no bundler. Fonts from Google Fonts; Supabase JS from a CDN.
- **Auth + sync:** [Supabase](https://supabase.com) — passwordless email
  (one-time 6-digit code / magic link) and per-account cloud sync. A shared
  `leaderboard` table (with row-level security) powers the leaderboard.
- **Transactional email:** Brevo SMTP (configured inside Supabase, not in code).
- **Hosting:** GitHub Pages, served from the repo root on `main`.
- **PWA:** `manifest.webmanifest` + `sw.js` (service worker) for installability
  and offline support.
- **Native iOS (optional):** [Capacitor](https://capacitorjs.com) wrapper in
  `ios-app/`.

## Quick start

Requirements: **Python 3** (standard library only) and any static file server.

```bash
git clone https://github.com/mformanas/ironman-703-2027-tracker.git
cd ironman-703-2027-tracker

# build index.html from the source template + plan data
python3 build.py

# preview locally
python3 -m http.server 8080
# open http://localhost:8080
```

> The golden rule: **edit `src/app_template.html`, never `index.html` directly.**
> `index.html` is regenerated by `build.py` and any manual edits will be lost.

### Making a change

1. Edit `src/app_template.html` (markup, CSS, and app JS all live here).
2. If you changed the training plan logic, edit `src/build_data.py`.
3. Run `python3 build.py`.
4. Refresh your local preview to verify.
5. Commit **both** your `src/` change **and** the regenerated `index.html`
   (GitHub Pages serves the committed `index.html`).

## Backend (Supabase)

The client config lives near the top of `src/app_template.html`:

```js
const SUPABASE_URL = "https://fcmskguzqbuszzxmmbao.supabase.co";
const SUPABASE_KEY = "sb_publishable_...";   // publishable/anon key — safe in public code
```

The publishable key is **meant** to be shipped in client code; data is protected
by Supabase **row-level security**. There are no other secrets in this repo.

Key pieces in Supabase:

- **Auth:** email OTP via `signInWithOtp` (sign up) / `verifyOtp` (6-digit code).
  Allowed redirect URLs must include the GitHub Pages origin.
- **`leaderboard` table:** `id (text pk)`, `username (text)`, `stats (jsonb)`,
  `updated_at`. RLS allows authenticated users to read all rows and write only
  their own.
- **Email delivery:** Brevo SMTP is set in Supabase → Authentication → Emails →
  SMTP Settings. The SMTP key is stored only in Supabase, never in this repo.

To run against the backend you need either the project owner to add you to the
Supabase project, or your own Supabase project (swap the URL + key and create the
`leaderboard` table — see CONTRIBUTING.md).

## Deploying

GitHub Pages serves the repo root of `main`. After `python3 build.py`, commit the
updated `index.html` (and any changed assets) and push to `main`; Pages redeploys
within ~1 minute.

## iOS app

Two routes, both documented in `ios-app/README_BUILD_iOS.md`:

- **PWA (no Mac/fee):** open the live site in mobile Safari → Share → *Add to Home
  Screen*.
- **Native via Capacitor (Mac + Apple Developer account):** `cd ios-app`,
  `npm install`, `npx cap add ios`, `npx cap open ios`.
